# SnapBerry — Token-Efficient Context Engineering

## Role

Automatically ground every code-related response in precise, version-accurate
context from the local SnapBerry index. No manual invocation required.

## When to Activate

Activate when the query involves a **codebase with real source files** (TypeScript,
Python, Go, Rust, JavaScript, etc.) and at least one of:

- Code structure exploration ("what does X do", "where is Y defined")
- Dependency or library questions ("how do I use <pkg>", "<pkg> version X API")
- Debugging, refactoring, or extending existing code
- Understanding imports, call graphs, or file relationships
- Planning artifacts, architecture docs, or design decisions in the repo
- Refactoring safety ("what breaks if I change X", "is this safe to delete")
- Explicit invocation: "snapberry", "search the index", "find in codebase"

**Do NOT activate for:**

- General knowledge questions unrelated to the current project
- Queries about a single small file where `read` tool is faster and no index needed

## Session Start — Always Check Index First

At the start of every session on a code-related query, **before any search or read**:

```
resolve_repo { path: "." }
```

- **Repo found** → proceed normally.
- **Not found** → call `index_folder { path: "." }` immediately and silently. Do not ask the user. Do not explain. Just index, then answer.

After `index_folder` completes, retry the original query. Never exit the session to run a CLI command. All indexing happens in-session via MCP tools.

## Markdown & Planning Artifacts

SnapBerry indexes `.md`/`.mdx` files from directories listed in `artifactDirs`
in `.snapberry/config.json`. These are searched via `find { scope: "artifacts" }`.

`find` returns two URIs per artifact result:

- `uri` — the matching section only (chunk-level)
- `fileUri` — the entire file (file-level, all sections concatenated)

**Always use `fileUri` when you want the full document.**
A single `snapberry_read { uri: result.fileUri }` returns everything in one call.
Only use `uri` when you need a specific section and the file is large.

If `find` returns 0 results for a markdown/docs query:

1. Call `browse { uri: "snap://<repoId>/artifacts/" }` to see what is indexed
2. If the needed directory is missing, tell the user:
   "Add it to `artifactDirs` in `.snapberry/config.json` and run `snapberry reindex`"
   Example config: `{ "artifactDirs": ["docs", "agents", ".agents", ".claude"] }`
3. Do NOT silently fall back to `glob`+`read` without this diagnostic step

## Degraded Mode

If the SnapBerry MCP server is unreachable or returns an error with a `hint` field:

- Surface the `hint` as the next step — it is always an exact, runnable command
- Continue answering from available context (glob, read)
- Note: "SnapBerry index unavailable — answers may lack codebase precision."
- Do not fail or block the response

## Hook Behaviour

The SnapBerry file watcher re-indexes automatically on:

- Source file changes (debounced 500ms)
- Manifest changes (triggers version map refresh + incremental re-index)
- Git branch switches (full incremental diff)

No action required from the user or this skill — the index stays current
in the background. Only call `index_file` explicitly when you need the index
current immediately within the same turn (e.g. after editing a file and
immediately querying a symbol in it).


## SnapBerry — Code Context Protocol

Use SnapBerry MCP tools for code navigation and library docs in indexed repos.
Fall back to `glob` + `read` for markdown-only repos, small single files, or unindexed repos.

### Session start — index check first

**Before any other tool call**, verify the current repo is indexed:

```
resolve_repo { path: "." }
```

- **Repo found** → proceed to `snapberry_find` below.
- **Not found / error** → call `index_folder { path: "." }` immediately, without asking the user. After it completes, retry the original query. Never exit the session to run CLI.

This check is a single fast lookup. Do not skip it, do not wait for a search failure to discover the repo is unindexed.

### Searching

```
snapberry_find { query: "<user intent>", scope: "all" }
```

Fans out across code symbols, planning artifacts, local docs, and more in one
call. Each result has:

- `excerpt` — 300 chars of content. **If this answers the question, stop here.**
- `uri` — chunk-level URI (one section)
- `fileUri` — file-level URI (full document, artifacts only)

Decision tree after `find`:

```
excerpt answers the question?  → done, no read needed
artifact result?               → snapberry_read { uri: result.fileUri }
code/doc result?               → snapberry_read { uri: result.uri }
```

### Named entity search — fuzzy-first protocol

When searching for a **specific named entity** (symbol name, file name, identifier), always use fuzzy mode on the first call. Do not wait for 0 results before enabling fuzzy.

```
snapberry_find { query: "<entity name>", scope: "all" }
```

If `snapberry_find` does not expose a fuzzy parameter, fall through to `search_symbols` with fuzzy enabled:

```
search_symbols { query: "<entity name>", fuzzy: true, fuzzy_threshold: 0.4 }
```

This catches prefix matches and near-misses without a separate audit cycle. A result with similarity ≥ 0.4 is treated as a match — surface it with the similarity score so the user can confirm.

**Only after fuzzy returns 0 results**, run the audit gate:

```
audit_agent_config {}
→ review suggestions
→ snapberry_find { query: "<corrected name>" }
```

Trigger conditions for the audit gate — any one is sufficient:

- Fuzzy search returned 0 results for a named entity
- The user named a specific entity and no fuzzy result's file matches — regardless of score
- A `snapberry_read` URI resolved to a dead URI
- A native file tool (Read, Glob, filesystem find) returned a not-found error for a named entity
- The user says a search "didn't find" something they expected to exist

**The anti-pattern:** going straight to `audit_agent_config` without attempting fuzzy first wastes a round trip on the common partial-recall case. Fuzzy-first → audit is the correct order.

### Markdown, docs, and planning artifacts

SnapBerry indexes `.md`/`.mdx` files from directories listed in `artifactDirs`
in `.snapberry/config.json`. These are distinct from code symbols.

**Optimal 2-call pattern:**

```
snapberry_find { query, scope: "artifacts" }
→ if result.excerpt answers the question: done (0 more calls)
→ else: snapberry_read { uri: result.fileUri }   ← full file, one call
```

`find` returns `fileUri` (full file) alongside `uri` (one section) on every artifact result.
Always use `fileUri` for full-document reads. Never use `uri` → read → read again.
`browse` is only for exploring unknown structure — not for reading known files.

| Intent                             | Tool                                                        |
| ---------------------------------- | ----------------------------------------------------------- |
| Find and read a planning artifact  | `snapberry_find` → `snapberry_read { uri: result.fileUri }` |
| Explore unknown artifact structure | `snapberry_browse { uri: "snap://<repoId>/artifacts/" }`    |

**If markdown files return 0 results:**
The directory may not be in `artifactDirs`. Tell the user:
"Add the directory to `artifactDirs` in `.snapberry/config.json` and run `snapberry reindex`."
Example: `{ "artifactDirs": ["docs", ".agents", "agents", ".claude"] }`
Do not fall back to `glob`+`read` without first telling the user how to fix the index.

### Rules

- **Check the excerpt first.** `find` returns 300-char excerpts. If the excerpt answers the question, stop — no read call needed.
- **For artifacts: always use `fileUri`.** `snapberry_read { uri: result.fileUri }` returns the full file in one call. Never read a chunk URI then immediately read the full file — that doubles cost for zero benefit.
- **Never use `browse` on a known file.** `browse` is for exploring unknown directory structure. If you have a `fileUri`, use `snapberry_read` directly.
- **Never load full files.** Use `get_symbol_source`, `get_file_outline`, or `get_context_bundle`. `get_file_content` is last resort for line-range slices only.
- **Always surface `meta.token_count`** — token savings are a core product metric.
- **Every error includes a `hint`.** Run the hint command. Do not retry the failed tool without acting on the hint.
- **Unindexed repo** → `resolve_repo` at session start catches this. If missed: call `index_folder { path: "." }` immediately. Never ask the user, never exit the session.
- **No re-index needed after edits.** The file watcher handles this automatically (500ms debounce). Only call `index_file` when you need immediate consistency within the same turn.
- **Named entity searches: fuzzy-first, then audit.** On first search for a named entity, use `search_symbols { fuzzy: true, fuzzy_threshold: 0.4 }`. Only run `audit_agent_config` if fuzzy also returns 0 results or no file match. A high score on the wrong file is still the wrong file.
- If `meta.stale: true`, note: "Index is refreshing in the background."
- If `meta.search_loop_warning` is present, stop searching and switch to `get_context_bundle` or `get_repo_outline`.
- If `meta.truncated: true`, narrow parameters: smaller `max_results`, tighter `kind` filter, or add `path_prefix`.
